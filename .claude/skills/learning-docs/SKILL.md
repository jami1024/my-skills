# Learning Docs - 技术学习文档生成

融合三种文档方法论的精华，生成系统化、易于理解的技术学习文档。

## 设计理念

本 skill 融合了三种经过验证的文档方法论：

| 来源 | 贡献 | 在文档中的位置 |
|------|------|----------------|
| **Diataxis Tutorial** | 项目驱动、动手学习、"你应该看到"模式 | 主体（教程部分） |
| **Technical Writer** | 排错指南、FAQ、文档检查清单 | 附录 A/B |
| **Diataxis Framework** | 参考表、设计哲学解析 | 附录 C/D |

## 生成流程

### 第一步：生成学习规划

在生成具体文档之前，先为该主题创建学习规划文件（`{topic}-plan.md`）。

```markdown
# {主题} 学习规划

## 学习目标
用 1-2 句话说明学完后能做什么。

## 知识点清单

### 阶段一：{阶段名}（基础）
1. {知识点} — {一句话说明}
2. {知识点} — {一句话说明}

### 阶段二：{阶段名}（核心）
3. {知识点} — {一句话说明}
4. {知识点} — {一句话说明}

### 阶段三：{阶段名}（实践）
5. {知识点} — {一句话说明}

## 配套代码示例
- `examples/01-xxx/main.go` — {说明}
- `examples/02-xxx/main.go` — {说明}
```

### 第二步：生成学习文档

按照下面的文档结构模板生成主文档。

### 第三步：生成可运行示例代码

每个知识点配套一个可独立运行的示例程序，放在 `examples/` 目录下。

### 第四步：验证

- 所有示例代码必须能编译运行
- 文档中引用的输出必须与实际运行结果一致

## 写作原则

### 核心原则：动手学习

教程不是讲义。读者在每个时刻都应该在**做**某件事。

| 避免 | 推荐 |
|------|------|
| "接口是一种抽象机制..." | "创建一个文件，定义你的第一个接口：" |
| "HTTP 协议使用请求-响应模型..." | "运行这个命令发送你的第一个请求：" |
| "有多种方式可以实现..." | "把这段代码加到你的文件中：" |

### 原则一：一条路径，不给选择

教程中只提供一种方式。不要列出替代方案，不要说"你也可以用 X"。

### 原则二：每一步都有可见结果

每个操作之后，告诉读者他们**应该看到什么**。这是建立信心的关键。

```markdown
运行程序：
```bash
go run main.go
```

你应该看到：
```
小猫说：喵喵！ 🐱
小狗说：汪汪！ 🐕
```

恭喜！你的第一个接口已经工作了。
```

### 原则三：老师承担责任

如果读者失败了，是教程的错，不是读者的错。预判可能的问题并提前防范。

```markdown
> 如果你看到 "cannot find package" 错误，请确认你在项目根目录下运行了 `go mod init`。
```

### 原则四：用项目串联知识点

不要把知识点孤立地讲。设计一个贯穿全文的小项目（如"动物园管理程序"、"待办事项 CLI"），让每一步都在给这个项目增加功能。

### 原则五：正反对比，加深理解

在排错指南和 FAQ 中使用正反对比：

```markdown
// 错误写法
var a Animal
a.Speak()  // panic: nil pointer

// 正确写法
var a Animal = &Cat{Name: "小花"}
a.Speak()  // 正常运行
```

### 原则六：设计哲学放在附录

不要在教程主体中解释"为什么要这样设计"。主体专注于**做**，设计讨论放到附录 D 给想深入理解的读者。

### 原则七：参考表要简洁实用

附录 C 的参考表是为**工作时查阅**准备的，不是教学材料。保持简洁，每行一个条目。

## 语言风格

### 中文写作规范

- 使用简洁口语化的表述，避免学术腔
- 中英文之间加空格：`Go 语言`、`interface 接口`
- 代码注释使用中文
- 术语首次出现时标注英文：`接口（interface）`
- 使用"你"而不是"您"，保持亲切

### 代码规范

- 所有代码必须完整可运行，不省略 import
- 代码注释简洁明了
- 变量名使用目标语言的惯例（Go 用驼峰，Python 用下划线）
- 示例数据要有趣味性（用动物、食物等生活化的例子）

### 标题层级

- `#` 文档标题（仅一个）
- `##` 教程步骤 / 附录标题
- `###` 步骤内的子节 / 排错条目

## 质量检查清单

生成文档后，逐项确认：

- [ ] 有学习规划文件，列出所有知识点和学习顺序
- [ ] 开头说明了"你将构建什么"
- [ ] 有一个贯穿全文的项目
- [ ] 每一步都有完整可运行的代码
- [ ] 每一步都有"你应该看到"的预期输出
- [ ] 没有在教程中提供多种替代方案
- [ ] 预期输出与实际运行结果一致
- [ ] 附录 A 包含 3-5 个常见排错问题
- [ ] 附录 B 包含 5+ 个 FAQ
- [ ] 附录 C 包含速查参考表
- [ ] 附录 D 讨论了关键设计决策的"为什么"
- [ ] 所有示例代码能独立编译运行
- [ ] 用语亲切简洁，没有学术腔
