# Handoff Summary — 会话交接总结生成

基于当前完整对话历史，压缩生成新 AI / 新会话能立刻接手的"任务包"。

## 核心原则

1. **只使用对话中已有的信息**，严禁推断脑补
2. 缺信息就写 `⚠️ 信息缺失：……`
3. 偏向可执行、具体、有证据支撑；禁止模糊表述
4. 根据对话实际内容自动判断使用哪种模式

## 模式选择

用户触发此 skill 时，根据以下规则选择模式：

| 条件 | 模式 |
|------|------|
| 对话涉及代码编写、文件修改、bug 修复、项目开发 | **代码交接模式** |
| 用户明确说"代码交接"或"code handoff" | **代码交接模式** |
| 其他所有情况（策略讨论、学习、设计、规划等） | **通用交接模式** |
| 用户明确指定模式 | 按用户指定 |

如果不确定，直接问用户。

## 通用交接模式

适用于：技术讨论、策略规划、学习探索、创意设计、方案评估等非编码对话。

### 输出模板

```markdown
# 《{从对话中提炼的主题名}》交接总结

## 0. TL;DR
- 当前阶段/卡点：
- 核心未解决问题：

## 1. 背景与目标
- 为什么讨论这个（触发事件）：
- 最终要达成的结果：
- 成功标准（满足哪些条件算完成）：
- 约束与边界（时间/平台/权限/不可改动项等）：

## 2. 现状快照
- 已确认的事实：
- 仍不确定/需验证的点：
- 当前的核心问题或现象：
- 影响范围（对用户/项目/决策的影响）：

## 3. 已讨论/已尝试的内容
按时间线或模块列出，每条包含：
- 做了什么 → 结论/结果 → 为什么行或不行

## 4. 关键决策记录
每条包含：
- 决策：选了什么/排除了什么
- 理由：核心权衡是什么
- 后续影响：对接下来的限制或方向

## 5. 核心结论与证据
每条结论附上支撑依据（原话、数据、推理链）

## 6. 问题清单（按优先级）
- P0/P1/P2 标注
- 问题描述
- 阻塞原因
- 建议的最小验证动作

## 7. 下一步行动计划（最多 5 条，必须可操作）
- [ ] 动作 + 预期结果 + 备选/兜底方案

## 8. 给新 AI 的最小启动包
- 必须知道的 5 条事实：
- 必须先确认的 3 件事：
- 绝对不要踩的 3 个坑：
```

## 代码交接模式

适用于：编码任务、bug 修复、功能开发、重构、项目搭建等涉及代码变更的对话。

在通用模式基础上，增加代码专属上下文，确保新会话能精准接手代码工作。

### 输出模板

```markdown
# 《{从对话中提炼的主题名}》代码交接总结

## 0. TL;DR
- 当前阶段/卡点：
- 核心未解决问题：
- 代码状态：✅ 可运行 / ⚠️ 部分完成 / ❌ 有阻塞

## 1. 任务背景
- 任务描述（一句话）：
- 最终要达成的结果：
- 成功标准：
- 约束（技术栈/版本/不可改动项）：

## 2. 项目上下文
- 项目名称/路径：
- 技术栈：
- 关键依赖及版本：
- 项目结构概要（只列与任务相关的部分）：
```
相关目录/
├── ...
└── ...
```

## 3. 代码变更记录
按时间线列出所有已完成的变更：

### 变更 1：{描述}
- 文件：`path/to/file`
- 改了什么：
- 为什么这样改：
- 状态：✅ 完成 / ⚠️ 需要调整

### 变更 2：{描述}
...

## 4. 当前代码状态
- 哪些文件是新增的：
- 哪些文件被修改了：
- 哪些文件被删除了：
- 未提交的变更（如有）：
- 当前分支：

## 5. 已知问题与技术债务
每条包含：
- 问题描述
- 所在文件/行号
- 严重程度：P0/P1/P2
- 临时方案（如有）

## 6. 测试状态
- 已通过的测试：
- 失败的测试：
- 未覆盖的场景：
- 运行测试的命令：

## 7. 关键决策记录
每条包含：
- 决策：选了什么方案/排除了什么
- 理由：核心权衡
- 代码影响：涉及哪些文件/模块

## 8. 阻塞问题（按优先级）
- P0/P1/P2 标注
- 问题描述
- 错误信息/日志（如有）
- 已尝试的解决方案
- 建议的下一步排查方向

## 9. 下一步行动计划（最多 5 条）
- [ ] 动作 + 涉及文件 + 预期结果 + 验证方式

## 10. 给新 AI 的最小启动包
- 必须知道的 5 条事实：
- 接手后第一步应该做什么：
- 必须先确认的 3 件事（建议先运行的命令）：
- 绝对不要踩的 3 个坑：
```

## 生成指南

### 信息提取原则

1. **逐条回溯对话**，不要只看最后几轮
2. 区分"已确认的事实"和"讨论中的假设"
3. 用户否定过的方案也要记录（标注为已排除）
4. 错误信息、日志输出要原样保留，不要改写

### 质量检查

生成后自检：
- [ ] 每条结论都有对话中的依据支撑吗？
- [ ] 有没有脑补对话中不存在的信息？
- [ ] 新 AI 读完能立刻开始工作吗？
- [ ] 行动计划是否具体到可执行？
- [ ] （代码模式）文件路径和代码片段是否准确？

### 长度控制

- 简短对话（< 10 轮）：精简输出，跳过无内容的章节
- 中等对话（10-30 轮）：标准输出
- 长对话（> 30 轮）：完整输出，确保不遗漏关键信息
- 空章节直接省略，不要留空模板
