# Learning Docs - 技术学习文档生成

融合三种文档方法论的精华，生成系统化、易于理解的技术学习文档。

## 设计理念

本 skill 融合了三种经过验证的文档方法论：

| 来源 | 贡献 | 在文档中的位置 |
|------|------|----------------|
| **Diataxis Tutorial** | 项目驱动、动手学习、"你应该看到"模式 | 主体（教程部分） |
| **Technical Writer** | 排错指南、FAQ、文档检查清单 | 附录 A/B |
| **Diataxis Framework** | 参考表、设计哲学解析 | 附录 C/D |

## 生成流程

### 第一步：生成学习规划

在生成具体文档之前，先为该主题创建学习规划文件（`{topic}-plan.md`）。

```markdown
# {主题} 学习规划

## 学习目标
用 1-2 句话说明学完后能做什么。

## 知识点清单

### 阶段一：{阶段名}（基础）
1. {知识点} — {一句话说明}
2. {知识点} — {一句话说明}

### 阶段二：{阶段名}（核心）
3. {知识点} — {一句话说明}
4. {知识点} — {一句话说明}

### 阶段三：{阶段名}（实践）
5. {知识点} — {一句话说明}

## 配套代码示例
- `{topic}-examples/01-xxx/main.go` — {说明}
- `{topic}-examples/02-xxx/main.go` — {说明}
```

### 第二步：生成学习文档

按照下面的文档结构模板生成主文档。

### 第三步：生成可运行示例代码

每个知识点配套一个可独立运行的示例程序，放在 `examples/` 目录下。

### 第四步：验证

- 所有示例代码必须能编译运行
- 文档中引用的输出必须与实际运行结果一致

## 写作原则

### 核心原则：动手学习

教程不是讲义。读者在每个时刻都应该在**做**某件事。

| 避免 | 推荐 |
|------|------|
| "接口是一种抽象机制..." | "创建一个文件，定义你的第一个接口：" |
| "HTTP 协议使用请求-响应模型..." | "运行这个命令发送你的第一个请求：" |
| "有多种方式可以实现..." | "把这段代码加到你的文件中：" |

### 原则一：一条路径，不给选择

教程中只提供一种方式。不要列出替代方案，不要说"你也可以用 X"。

### 原则二：每一步都有可见结果

每个操作之后，告诉读者他们**应该看到什么**。这是建立信心的关键。

```markdown
运行程序：
```bash
go run main.go
```

你应该看到：
```
小猫说：喵喵！ 🐱
小狗说：汪汪！ 🐕
```

恭喜！你的第一个接口已经工作了。
```

### 原则三：老师承担责任

如果读者失败了，是教程的错，不是读者的错。预判可能的问题并提前防范。

```markdown
> 如果你看到 "cannot find package" 错误，请确认你在项目根目录下运行了 `go mod init`。
```

### 原则四：用项目串联知识点

不要把知识点孤立地讲。设计一个贯穿全文的小项目（如"动物园管理程序"、"待办事项 CLI"），让每一步都在给这个项目增加功能。

### 原则五：正反对比，加深理解

在排错指南和 FAQ 中使用正反对比：

```markdown
// 错误写法
var a Animal
a.Speak()  // panic: nil pointer

// 正确写法
var a Animal = &Cat{Name: "小花"}
a.Speak()  // 正常运行
```

### 原则六：设计哲学放在附录

不要在教程主体中解释"为什么要这样设计"。主体专注于**做**，设计讨论放到附录 D 给想深入理解的读者。

### 原则七：参考表要简洁实用

附录 C 的参考表是为**工作时查阅**准备的，不是教学材料。保持简洁，每行一个条目。

## 语言风格

### 中文写作规范

- 使用简洁口语化的表述，避免学术腔
- 中英文之间加空格：`Go 语言`、`interface 接口`
- 代码注释使用中文
- 术语首次出现时标注英文：`接口（interface）`
- 使用"你"而不是"您"，保持亲切

### 去 AI 痕迹写作规范

生成的文档必须读起来像有经验的工程师写的，不像 AI 输出。遵循以下规则：

#### 规则一：禁用 AI 高频词

以下词汇在 AI 输出中出现频率异常高，禁止使用：

| 禁用 | 替代 |
|------|------|
| 此外 | 直接写下一句，不加连接词 |
| 至关重要 | 重要 / 关键 / 删掉 |
| 深入探讨/深入了解 | 看看 / 了解 / 学习 |
| 充满活力的 | 删掉，用具体描述 |
| 不断演变的格局 | 删掉 |
| 值得注意的是 | 直接陈述 |
| 无缝/无缝衔接 | 删掉，或用具体描述说明怎么衔接 |
| 强调/彰显/凸显 | 说明 / 体现 |
| 奠定基础/奠定坚实基础 | 是……的基础 / 删掉 |

#### 规则二：删除填充短语

直接说事，不铺垫：

| 删掉 | 改为 |
|------|------|
| "为了实现这一目标" | "为了" |
| "在这个时间点" | "现在" |
| "值得注意的是数据显示" | "数据显示" |
| "系统具有处理的能力" | "系统可以处理" |
| "让我们来看看" | 直接给代码 |
| "接下来我们将" | 直接进入下一步 |

#### 规则三：禁止三段式堆砌

AI 喜欢把东西凑成三个。如果三个并列项中有一个是凑数的，删掉它。

```markdown
# ❌ AI 味
通过这个教程，你将掌握灵活、高效、优雅的编程方式。

# ✅ 人话
这个教程教你用接口写解耦的代码。
```

#### 规则四：禁止夸大意义

不要给普通技术点加戏。说它是什么就够了，不用说它"多重要"。

```markdown
# ❌ AI 味
接口是 Go 语言中至关重要的特性，它为构建灵活的软件架构奠定了坚实的基础。

# ✅ 人话
Go 用接口做抽象。你定义一组方法签名，任何实现了这些方法的类型都自动满足接口。
```

#### 规则五：禁止通用积极结论

不要用"激动人心的旅程""光明的未来"之类的话收尾。结尾给具体的下一步行动。

```markdown
# ❌ AI 味
恭喜你完成了这段精彩的学习旅程！接口的世界充满无限可能，期待你创造更多精彩！

# ✅ 人话
你现在会用接口了。下一步可以看看 io.Reader 和 io.Writer，它们是 Go 标准库里用得最多的接口。
```

#### 规则六：变化句式节奏

连续三句话不能长度和结构相同。长短交替，偶尔用一个短句打断节奏。

```markdown
# ❌ 机械节奏
接口定义了一组方法签名。任何类型只要实现了这些方法就满足该接口。这种机制称为隐式实现。

# ✅ 有节奏
接口就是一组方法签名。实现了这些方法？恭喜，你自动满足这个接口——不用写 implements。Go 管这叫隐式实现。
```

#### 规则七：用具体替代抽象

不说"强大的功能"，说具体是什么功能。不说"广泛应用"，说具体在哪用。

```markdown
# ❌ 抽象
channel 是 Go 并发编程的强大工具，在各种场景中有广泛应用。

# ✅ 具体
channel 用来在 goroutine 之间传数据。比如一个 goroutine 抓网页，另一个处理结果，中间用 channel 连起来。
```

### 代码规范

- 所有代码必须完整可运行，不省略 import
- 代码注释简洁明了
- 变量名使用目标语言的惯例（Go 用驼峰，Python 用下划线）
- 示例数据要有趣味性（用动物、食物等生活化的例子）

### 标题层级

- `#` 文档标题（仅一个）
- `##` 教程步骤 / 附录标题
- `###` 步骤内的子节 / 排错条目

## 质量检查清单

生成文档后，逐项确认：

### 结构完整性
- [ ] 有学习规划文件，列出所有知识点和学习顺序
- [ ] 开头说明了"你将构建什么"
- [ ] 紧接着有"你将学习到什么"，列出核心知识点
- [ ] 有一个贯穿全文的项目
- [ ] 每一步都有完整可运行的代码
- [ ] 每一步都有"你应该看到"的预期输出
- [ ] 每一步末尾有"这一步你学到了"小结
- [ ] 没有在教程中提供多种替代方案
- [ ] 预期输出与实际运行结果一致
- [ ] 附录 A 包含 3-5 个常见排错问题
- [ ] 附录 B 包含 5+ 个 FAQ
- [ ] 附录 C 包含速查参考表
- [ ] 附录 D 讨论了关键设计决策的"为什么"
- [ ] 所有示例代码能独立编译运行

### 去 AI 痕迹检查
- [ ] 全文不含"此外""至关重要""深入探讨""充满活力""不断演变的格局"等 AI 高频词
- [ ] 没有"让我们来看看""接下来我们将"等填充铺垫
- [ ] 并列项没有刻意凑成三个
- [ ] 没有夸大技术的意义（"奠定坚实基础""至关重要的特性"）
- [ ] 结尾给具体下一步行动，没有"精彩旅程""无限可能"之类的空话
- [ ] 连续三句话的长度和结构不雷同
- [ ] 描述用具体细节，不用"强大""广泛""丰富"等抽象形容词
- [ ] 读出来像有经验的工程师在聊天，不像教科书或新闻稿
