# Project Archive — 项目技术归档

将代码和架构信息"冻结"成高密度技术文档，用于团队知识共享、人员交接和版本发布记录。

## 核心定位

这不是写给最终用户的说明书，而是写给**接手的开发者**的"大脑转储 (Brain Dump)"。

重点在于：
- **What** — 做了什么
- **How** — 怎么实现的
- **Why** — 为什么这么设计

## 与 handoff-summary 的区别

| 维度 | handoff-summary | project-archive |
|------|----------------|-----------------|
| 粒度 | 会话级（对话上下文） | 项目/功能级（代码 + 架构） |
| 触发时机 | 对话快结束时 | 功能完成 / 版本发布 / 代码变更后 |
| 信息来源 | 对话历史 | 对话历史 + 实际代码扫描 |
| 输出深度 | 轻量快速 | 深度完整 |
| 侧重点 | 决策、卡点、下一步 | 架构、实现、契约、技术债务 |

## 三种模式

根据用户触发词自动选择模式：

| 模式 | 触发词 | 场景 | 输出 |
|------|--------|------|------|
| **项目归档** | "项目归档"、"阶段归档"、"版本归档" | 开发阶段完成，全局归档 | 新建完整归档文档 |
| **功能归档** | "功能归档 [X]"、"补功能文档 [X]" | 某个功能缺文档，需要深度补齐 | 新建功能级归档文档 |
| **文档更新** | "更新文档"、"同步文档" | 代码改了，已有文档需要同步 | 增量更新已有文档 |

如果不确定用户要哪种模式，直接问。

---

## 通用执行步骤

三种模式共享前两步：

### 第一步：全局扫描

1. 回顾当前会话中修改/新增/删除的所有文件
2. 扫描项目结构，识别核心模块和关键文件
3. 检查 git 状态（当前分支、未提交变更、最近提交记录）
4. 识别项目技术栈（语言、框架、构建工具、包管理器）

### 第二步：确认范围

向用户确认：
1. 归档/更新的目标是什么？（项目阶段 / 具体功能 / 指定文档）
2. 文件放在哪个目录？（默认建议 `agent-doc/reference/`）
3. 有没有需要特别强调的设计决策或遗留问题？

## 模式二：功能归档

适用于某个具体功能缺少文档，需要深度补齐。重点在于区分"原有实现"和"本次变更"。

### 执行

1. **锁定功能范围**：明确目标功能的入口、核心逻辑、数据层
2. **代码为准**：必须先通过搜索和读取源码获取信息，严禁只靠对话历史生成结论
3. **区分新旧**：先说明原有实现，再说明本次变更，最后说明对后续的影响

命名格式同项目归档：`v{版本}_{功能模块}_context.md`

### 文档模板

# [{版本/功能}] 技术上下文归档

> 归档范围：{功能名称}

## 模式三：文档更新

适用于代码变更后，同步更新已有的归档文档。确保文档反映当前代码的真实状态 (Source of Truth)。

### 执行

1. **锁定目标文档**：确认要更新的文档路径。如果用户未指定，根据当前修改的代码推断最相关的文档
2. **读取原文档**：读取目标文档全部内容
3. **代码比对**：分析当前会话中的代码变更（git diff 或最近修改的文件），与原文档描述进行对比
4. **执行更新**：直接修改该文档

### 更新操作类型

| 操作 | 条件 | 做法 |
|------|------|------|
| **新增** | 有新功能/新接口 | 在文档对应位置添加章节 |
| **修改** | 逻辑变了 | 更新描述和流程图 |
| **删除/标记** | 功能被移除或废弃 | 删除或标记为 `(Deprecated)` |
| **纠错** | 文档描述与代码不符 | 以代码为准修正 |

### 更新原则

- **保持结构**：尽量保持原文档的目录结构，除非原结构已完全不适用
- **高亮变更**：重要的逻辑变更使用 `> ⚡ 变更` callout 强调
- **Mermaid 更新**：如果代码逻辑改变影响了流程图，必须重新生成
- **版本记录**：在文档尾部追加更新记录：`Updated on {Date} for {Feature/Reason}`

## 生成指南

### 语言无关原则

此 skill 适用于任何技术栈。根据项目实际情况调整术语：

| 技术栈 | 组件路径示例 | 数据结构示例 | 构建命令示例 |
|--------|-------------|-------------|-------------|
| Go | `internal/service/user.go` | struct 定义 | `go build ./...` |
| Python | `src/services/user_service.py` | Pydantic model | `pytest` / `uvicorn` |
| Java | `src/main/java/.../Service.java` | Entity/DTO class | `mvn package` |
| TypeScript | `src/features/user/api.ts` | interface/type | `npm run build` |
| Rust | `src/handlers/user.rs` | struct + impl | `cargo build` |

### 信息提取原则

1. **代码优先**：优先从实际代码中提取信息，而非仅依赖对话记忆
2. **准确性**：不确定的逻辑标记为 `⚠️ 待确认`，不要编造
3. **代码引用**：关键逻辑必须包含文件路径和行号引用
4. **简洁性**：使用技术术语，避免废话，面向开发者写作

### 质量检查

生成后自检：
- [ ] 架构图是否准确反映了实际代码结构？
- [ ] 所有文件路径引用是否存在且正确？
- [ ] 接口契约是否与代码一致？
- [ ] 技术债务是否从代码中的 TODO 和已知问题中完整提取？
- [ ] 新开发者读完能否理解系统并开始工作？
- [ ] （功能归档）原有实现和本次变更是否清晰区分？
- [ ] （文档更新）更新内容是否与代码变更一致？

### 空章节处理

- 没有数据库变更？省略相关小节
- 没有新增依赖？省略相关小节
- 没有技术债务？注明"当前无已知技术债务"
- 不要留空模板占位
