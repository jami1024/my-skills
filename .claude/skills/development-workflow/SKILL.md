# 软件开发工作流程

语言无关的通用开发流程和最佳实践。与具体技术栈的 skill（fastapi/golang/react-best-practices）协同使用。

编写正式文档（需求、设计、ADR）时，配合 **doc-coauthoring skill** 的结构化协作流程和标准模板。

---

## 核心开发流程

```
1. 需求引导      ──  理解目的、约束、成功标准
      ↓
2. 设计方案      ──  多方案对比、增量展示、用户审查
      ↓
3. 实施规划      ──  细粒度任务分解、禁止占位符
      ↓
4. 代码实现      ──  先设计后实现、小步提交
      ↓
5. 代码审查      ──  功能/可读/安全/性能多维审查
      ↓
6. 测试验证      ──  测试金字塔、关键路径优先
```

---

<HARD-GATE>
在向用户展示设计方案并获得批准之前，不得编写任何代码、搭建任何项目骨架，也不得采取任何实施行动。此规则适用于所有项目，无论其看起来多么简单。
</HARD-GATE>

### 反模式："这太简单了，不需要设计"

每个项目都必须走步骤 1-2。一个 TODO 列表、一个单函数工具、一个配置修改——统统需要。"简单"项目恰恰是未经检验的假设导致最多返工的地方。设计可以很简短（真正简单的项目只需几句话），但你**必须**展示设计并获得用户批准。

### 判断是否需要完整流程

- **需要完整步骤 1-3**：新功能、多模块影响、架构变更、需要评审的改动
- **可简化步骤 1-2（但不跳过）**：简单 Bug 修复、单文件小改动——仍需确认理解和方案

## 1️⃣ 需求引导

### 探索项目上下文

在提问之前，先了解当前状态：

- 查看项目文件结构、文档、README
- 检查近期提交记录，了解项目动向
- 阅读相关代码，理解现有架构和模式
- 如果是现有代码库，**跟随已有模式**

### 评估范围

在问详细问题之前，先评估范围：

- 如果请求描述了**多个独立子系统**（例如"构建一个包含聊天、文件存储、计费和分析的平台"），立即标记
- 帮用户拆解为独立子项目，每个子项目独立走完设计 → 规划 → 实施的完整周期

### 提问原则

| 原则 | 说明 |
|------|------|
| **每次只问一个问题** | 不要用多个问题压垮用户 |
| **优先使用选择题** | 比开放式问题更容易回答，降低认知负担 |
| **一条消息只含一个问题** | 需要深入探索的话题拆成多条消息 |
| **聚焦三个维度** | 目的（为什么做）、约束（不能做什么）、成功标准（怎么算做好了） |

### 提问聚焦方向

- **目的**：你想解决什么问题？谁来用？用在什么场景？
- **约束**：技术栈限制？性能要求？兼容性要求？时间限制？
- **成功标准**：怎么判断这个功能做好了？用户看到什么就算成功？
- **边界**：哪些是不需要的？明确"不做什么"和"做什么"同样重要

### 用户故事格式（可选）

对于需要文档化的需求，可使用用户故事格式：

```
作为 <角色>
我想要 <功能>
以便于 <价值>

验收标准：
- [ ] 具体标准 1
- [ ] 具体标准 2
```

## 2️⃣ 设计方案

### 多方案探索

理解需求后，**不要直接给出单一方案**：

1. **提出 2-3 种不同方案**，每种附带：
   - 实现思路概述
   - 优点与缺点
   - 适用场景和复杂度评估

2. **给出推荐**，先展示推荐方案，说明为什么推荐这个而非其他

3. **等待用户选择**后再深入细化

### 增量展示设计

一次展示设计的一个部分，不要一股脑全部甩出来：

- **根据复杂度调整篇幅**：简单的几句话，复杂的 200-300 字
- **每段之后都问"到目前为止是否正确"**
- 覆盖以下主题（按需）：架构概览、核心组件及职责、数据流向、错误处理策略、测试策略

如果用户不同意：回到对应环节澄清分歧，用户的判断优先。

### 架构决策记录（ADR）

对重要的技术决策（技术栈选择、架构模式、影响长期维护的决策），使用 ADR 记录：

```markdown
# ADR-001: 选择 PostgreSQL 作为主数据库

## 状态：已接受

## 背景
需要为新项目选择数据库，要求支持复杂查询、事务和高并发。

## 决策
选择 PostgreSQL。

## 原因
1. 强大的 ACID 事务支持
2. 丰富的数据类型（JSON、数组等）
3. 优秀的性能和可扩展性

## 后果
- 正面：稳定可靠，功能丰富
- 负面：相比 NoSQL 在某些场景下性能稍差
```

### 规格自审

设计展示完毕后，以全新视角审视：

| 检查项 | 具体做什么 |
|-------|-----------|
| **占位符扫描** | 搜索 "TBD"、"TODO"、"待定"。找到就修复。 |
| **内部一致性** | 各部分是否矛盾？架构和功能描述是否匹配？ |
| **范围检查** | 是否聚焦到可以写出一个实施计划的程度？ |
| **歧义检查** | 是否有需求可以被两种方式解读？选定一种并明确。 |
| **YAGNI 检查** | 是否包含了用户未要求的特性？砍掉。 |

发现问题就**直接修复**。

### 用户审查

自审通过后，请用户确认设计方案：

> "设计方案已整理完毕。请确认是否有需要调整的地方，确认无误后我会开始制定实施计划。"

**只标记会导致实施问题的内容**（缺失关键信息、内部矛盾、严重歧义）。不纠结措辞风格和格式偏好。

等待用户回复。用户批准后进入步骤 3。

## 3️⃣ 实施规划

### 基本原则

- **零上下文假设**：假设执行者对代码库不熟悉，写清一切
- **DRY**：不要重复自己
- **YAGNI**：不要实现不需要的东西
- **TDD 优先**：先写测试 → 验证失败 → 写实现 → 验证通过
- **频繁提交**：每个有意义的步骤完成后提交

### 文件结构规划

在定义任务之前，先列出将要创建或修改的文件及其职责：

- 每个文件应有一个清晰的职责
- 优先使用小而聚焦的文件
- 一起变化的文件应放在一起，按职责而非技术层级拆分

### 细粒度任务分解

每一步是一个动作（2-5 分钟可完成）：

| 步骤类型 | 示例 |
|---------|------|
| 写测试 | "编写 `test_user_login` 测试用例" |
| 验证失败 | "运行测试，确认因 `function not defined` 而失败" |
| 写实现 | "实现 `user_login` 函数的最小代码" |
| 验证通过 | "运行测试，确认通过" |
| 提交 | "提交当前修改" |

### 任务结构模板

````markdown
### 任务 N：[组件名称]

**文件：**
- 创建：`exact/path/to/file.py`
- 修改：`exact/path/to/existing.py:123-145`
- 测试：`tests/exact/path/to/test.py`

- [ ] **步骤 1：编写失败测试**

```python
def test_specific_behavior():
    result = function(input)
    assert result == expected
```

- [ ] **步骤 2：运行测试验证失败**

运行：`pytest tests/path/test.py::test_name -v`
预期：FAIL

- [ ] **步骤 3：编写最小实现**

```python
def function(input):
    return expected
```

- [ ] **步骤 4：运行测试验证通过**

运行：`pytest tests/path/test.py::test_name -v`
预期：PASS

- [ ] **步骤 5：提交**

```bash
git add tests/path/test.py src/path/file.py
git commit -m "feat: 添加具体功能"
```
````

### 禁止占位符

以下是**计划失败**的标志——绝不允许出现：

| 禁止内容 | 说明 |
|---------|------|
| "TBD"、"TODO"、"后续实现"、"待补充" | 必须现在就写清楚 |
| "添加适当的错误处理" | 必须写出具体的错误处理代码 |
| "添加验证" / "处理边界情况" | 必须写出具体的验证逻辑 |
| "为上述代码编写测试"（不含测试代码） | 必须包含完整的测试代码 |
| "类似任务 N"（不重复代码） | 必须重复写出完整代码 |
| 描述做什么但不展示怎么做 | 代码步骤必须包含代码块 |

### 计划自审

| 检查项 | 具体做什么 |
|-------|-----------|
| **规格覆盖度** | 逐条检查设计中的每个需求，是否都有对应任务？ |
| **占位符扫描** | 搜索"禁止占位符"列表中的所有模式。 |
| **类型一致性** | 后面任务中的类型、方法签名是否与前面定义的一致？ |

发现问题就**直接修复**。缺少对应任务就**补充**。

### 执行交接

> "实施计划已制定完毕。你可以选择：
>
> 1. **按计划执行** — 我逐个任务执行，每个任务完成后汇报进度
> 2. **先审阅计划** — 你先看看计划，有修改意见再告诉我
>
> 选哪种？"

## 4️⃣ 代码实现

先设计后实现，小步提交（每次可运行），发现坏味道立即重构，至少写单元测试。

代码质量：命名清晰、函数 < 50 行、注释解释"为什么"而非"是什么"、DRY、错误处理完善。

### Git 提交规范（Conventional Commits）

```
<type>(<scope>): <subject>
```

Type：`feat` 新功能 | `fix` 修复 | `docs` 文档 | `refactor` 重构 | `perf` 性能 | `test` 测试 | `chore` 构建

示例：`feat(user): 添加用户批量导入功能`

---

## 5️⃣ 代码审查

通过 Code Review 发现问题、提高质量、分享知识。

### 审查清单

| 维度 | 检查要点 |
|------|---------|
| 功能性 | 需求实现？边界条件？错误处理？潜在 Bug？ |
| 可读性 | 易于理解？命名清晰？必要注释？结构清晰？ |
| 可维护性 | SOLID 原则？无重复？函数不过长？易扩展？ |
| 性能 | 性能问题？查询优化？内存泄漏？不必要计算？ |
| 安全性 | SQL 注入？XSS？敏感信息加密？权限检查？ |
| 测试 | 有单元测试？覆盖率足够？测试边界条件？ |

### Review 礼仪

- 审查者：提建设性意见（解释为什么），区分"必须改"和"建议改"，认可好代码
- 被审查者：开放心态，主动解释复杂逻辑，及时回应

---

## 6️⃣ 测试验证

### 测试金字塔

单元测试（大量，70-80% 覆盖）→ 集成测试（适量，关键路径）→ E2E 测试（少量，核心业务流程）

### 测试策略

关键路径优先 → 边界条件 → 错误场景 → 性能敏感功能 → Bug 修复后回归测试

### 命名规范

```
test_<功能>_<场景>_<预期结果>

示例：
test_create_user_with_valid_data_succeeds()
test_create_user_with_duplicate_email_fails()
```

## 🔀 Git 分支策略

### GitHub Flow（推荐中小型项目）

```
main ─────────────────────────────────────────→
  └── feature/add-user-import ──── PR ──→ merge
  └── fix/login-error ───────────── PR ──→ merge
```

**规则**：
1. `main` 分支始终可部署
2. 从 `main` 创建功能分支，命名：`feature/xxx`、`fix/xxx`、`refactor/xxx`
3. 通过 Pull Request 合并，必须通过 CI 和 Code Review
4. 合并后立即部署

### Git Flow（适合版本发布制项目）

```
main ────────────────────────── tag v1.0 ── tag v1.1 ──→
  └── develop ────────────────────────────────────────→
       └── feature/xxx ──→ merge to develop
       └── release/1.1 ──→ merge to main + develop
  └── hotfix/critical-bug ──→ merge to main + develop
```

**何时使用 Git Flow**：
- 有明确版本发布周期的项目
- 需要同时维护多个版本
- 需要 hotfix 机制

## 🚀 CI/CD 流水线

### 基础流水线结构

```
代码提交 → 代码检查 → 单元测试 → 构建 → 部署
   ↓          ↓          ↓        ↓       ↓
  Push      Lint      Test     Build   Deploy
          + Format   + Cover
```

### GitHub Actions 示例

```yaml
# .github/workflows/ci.yml
name: CI

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Setup
        uses: actions/setup-node@v4  # 或 setup-python、setup-go
      - name: Lint
        run: make lint

  test:
    runs-on: ubuntu-latest
    needs: lint
    steps:
      - uses: actions/checkout@v4
      - name: Setup
        uses: actions/setup-node@v4
      - name: Test
        run: make test
      - name: Upload coverage
        uses: codecov/codecov-action@v4

  build:
    runs-on: ubuntu-latest
    needs: test
    if: github.ref == 'refs/heads/main'
    steps:
      - uses: actions/checkout@v4
      - name: Build
        run: make build
```

### CI/CD 最佳实践

1. **快速反馈** - CI 流程控制在 10 分钟内
2. **并行执行** - lint 和 test 可以并行
3. **缓存依赖** - 使用 `actions/cache` 加速构建
4. **环境隔离** - staging 环境验证后再部署 production
5. **回滚机制** - 部署失败时能快速回滚
6. **密钥管理** - 使用 GitHub Secrets / Vault，不硬编码

## 核心设计原则

### YAGNI — 你不会需要它的

- 主动删减不必要的特性，不要为假设的未来需求设计
- 三行相似代码好过一个过早的抽象
- 只实现用户明确要求的功能

### 设计隔离与清晰性

- 把系统拆成更小的单元，每个单元只有一个清晰职责
- 通过明确定义的接口通信，可独立理解和测试
- 对每个单元，你应能回答：它做什么、怎么用它、它依赖什么

## 防绕过机制

以下想法意味着你正在找借口绕过步骤 1-2——**停下来**：

| 你的想法 | 现实 |
|---------|------|
| "这只是个简单修改" | 简单修改也是修改，走流程 |
| "我先了解下代码" | 了解代码是步骤 1 的一部分，不是跳过流程的理由 |
| "用户已经说了要什么" | 用户说了要什么 ≠ 设计已经完成 |
| "这个流程太重了" | 简单项目的设计只需几句话，流程不重 |
| "我先做一点再说" | 做之前先想清楚，这就是流程的意义 |
| "上次类似的不需要设计" | 每次都需要，无例外 |

---

## 常见陷阱

| 陷阱 | 正确做法 |
|------|---------|
| ❌ 跳过设计直接写代码 | ✅ 先需求引导和设计方案，获得用户批准再动手 |
| ❌ 忽略测试 | ✅ 至少写单元测试，Bug 修复后加回归测试 |
| ❌ 过度设计 | ✅ 够用就好，不为假设需求设计 |
| ❌ "以后再重构" | ✅ 发现坏味道立即重构 |
| ❌ 忽略 Code Review 反馈 | ✅ 反馈是学习机会，及时回应 |
| ❌ 代码和文档不同步 | ✅ 代码变更时同步更新文档 |
